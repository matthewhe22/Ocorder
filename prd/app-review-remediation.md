# PRD: App Review Remediation

**Date:** 2026-03-30
**Source:** Full application review (security, architecture, QA, legal, accessibility, SRE)
**Status:** Draft

---

## Background

A comprehensive multi-perspective review of the TOCS OCC Order Portal identified critical security vulnerabilities, architectural risks, zero test coverage, legal compliance gaps, and accessibility issues. This PRD captures all findings as actionable work items with acceptance criteria, prioritised for remediation.

---

## Critical — Must fix before next release

### SEC-1: Lock CORS to production origin

**Problem:** All serverless functions set `Access-Control-Allow-Origin: *`, exposing admin endpoints to cross-origin requests from any website.

**Acceptance criteria:**
- `cors()` in `api/_lib/store.js` sets `Access-Control-Allow-Origin` to the configured production domain only
- Preflight `OPTIONS` requests return the correct origin and method headers
- Requests from unlisted origins receive a 403

---

### SEC-2: Remove hardcoded default credentials

**Problem:** `Tocs@Vote` is committed to source as the default admin password in both `api/_lib/store.js` and `server.js`. It also doubles as the HMAC token signing key fallback, making it a dual-purpose critical secret.

**Acceptance criteria:**
- No default password exists in source code; startup fails loudly if `ADMIN_PASS` and `TOKEN_SECRET` env vars are absent
- `server.js` default credentials removed
- Git history reviewed; secret rotated in all environments

---

### SEC-3: Remove hardcoded financial details

**Problem:** Live BSB `033-065`, account number `522011`, and PayID `accounts@tocs.com.au` are committed to source in `api/_lib/store.js`.

**Acceptance criteria:**
- No financial details in source; payment details are only ever read from Redis config (set by admin via settings)
- Default config seed in store.js uses empty/placeholder values

---

### SEC-4: Fix test-sharepoint auth bypass

**Problem:** `api/config/test-sharepoint.js` checks only that an Authorization header is non-empty; it does not call `validToken()`. Any string passes.

**Acceptance criteria:**
- Endpoint calls `validToken()` from `api/_lib/store.js`
- Requests with an invalid or missing token receive a 401
- Behaviour is consistent with all other protected endpoints

---

### SEC-5: Validate and generate order IDs server-side

**Problem:** Order ID is generated by the frontend (`Math.random()`-based) and accepted verbatim by the server, allowing overwrite of existing orders.

**Acceptance criteria:**
- Server generates the order ID on `POST /api/orders`; any client-supplied ID is ignored
- ID uses `crypto.randomUUID()` or equivalent cryptographically secure source
- Server checks for ID collision before writing; returns 409 on conflict

---

### SEC-6: Rate limit login and order placement

**Problem:** No rate limiting on any endpoint; admin password is brute-forceable and order/email endpoints can be flooded.

**Acceptance criteria:**
- `POST /api/auth/login` returns 429 after 5 failed attempts within 15 minutes per IP
- `POST /api/orders` returns 429 after 10 requests within 60 seconds per IP
- Rate limit state stored in Redis with TTL; no in-memory state

---

### ARCH-1: Replace single Redis blob with per-entity keys

**Problem:** All plans, lots, and orders are stored in one JSON blob under `tocs:data`. Concurrent writes silently drop data; no recovery path exists.

**Acceptance criteria:**
- Orders stored individually under `tocs:order:{id}` keys
- Plans and lots stored under `tocs:plans` (separate from orders)
- All order-modifying endpoints use atomic Redis operations or optimistic locking (read version, write with check)
- Migration path from old `tocs:data` structure documented and tested
- Redis backup strategy defined (Upstash daily snapshots or equivalent)

---

## High — This sprint

### SEC-7: Sanitise error responses

**Problem:** Internal exception messages (potentially containing Redis URLs, stack traces) are returned directly to clients.

**Acceptance criteria:**
- All `catch` blocks return a generic message to the client (`"Internal server error"`)
- Full error details are logged server-side only (structured log with request ID)
- No `err.message` or `err.stack` in any HTTP response body

---

### SEC-8: Sanitise Content-Disposition filename

**Problem:** `order.lotAuthorityFile` (set from client-supplied filename) is embedded directly in the `Content-Disposition` header, enabling header injection.

**Acceptance criteria:**
- Filename is stripped of all characters outside `[a-zA-Z0-9._-]` before use in the header
- Header value is RFC 5987-encoded for non-ASCII filenames
- Unit test covers filenames containing `"`, `\r`, `\n`, and path separators

---

### SEC-9: Enable TLS certificate validation

**Problem:** `rejectUnauthorized: false` is set for both Redis and SMTP connections, disabling certificate verification.

**Acceptance criteria:**
- `rejectUnauthorized` removed (defaults to `true`) for both Redis and SMTP TLS connections
- Connections to Upstash and SMTP2GO verified to work with valid certificates
- Any required CA bundle is explicitly configured rather than bypassed

---

### SEC-10: Verify Stripe webhook signatures

**Problem:** `stripe-confirm` endpoint processes payment state transitions without verifying the Stripe webhook signature.

**Acceptance criteria:**
- Endpoint verifies the `Stripe-Signature` header using `stripe.webhooks.constructEvent()`
- Requests with invalid or missing signatures return 400 and are not processed
- `STRIPE_WEBHOOK_SECRET` added to env var documentation

---

### BE-1: Fix concurrent write race condition

**Problem:** All order-modifying endpoints use read-modify-write with no locking; concurrent Stripe retries can double-send emails and overwrite order state.

**Acceptance criteria:**
- `stripe-confirm` is idempotent via Redis `SET NX` or equivalent atomic check-and-set
- Status update endpoint uses optimistic locking (version field or Redis transaction)
- Integration tests confirm concurrent calls produce exactly one email and one state transition

---

### BE-2 & BE-3: Validate plans and lots input

**Problem:** Plans and lots are written to Redis without schema validation; malformed or malicious payloads accepted directly.

**Acceptance criteria:**
- Plans endpoint validates: required fields present, correct types, string length limits, no script content
- Lots import validates: array of objects with required fields, MIME-safe string values
- Validation errors return 400 with field-level detail
- Unit tests cover valid, invalid, and boundary inputs

---

### FE-1: Compute order total server-side

**Problem:** `order.total` is sent from the browser and used as the Stripe charge amount; customers can submit arbitrary amounts.

**Acceptance criteria:**
- `POST /api/orders` ignores any client-supplied total
- Server looks up product price from stored plan data and recomputes the total
- Recomputed total is used for the Stripe Checkout Session `unit_amount`
- Test confirms a tampered `total` in the request body does not affect the charge

---

### FE-2: Validate email address server-side

**Problem:** Customer email is accepted verbatim with no format validation; invalid addresses reach nodemailer.

**Acceptance criteria:**
- Server validates email format before storing or sending (RFC 5322 compliant regex or validator library)
- Invalid email returns 400 with a clear error message
- `to:` field in nodemailer call is always the server-validated value, not the raw input

---

### QA-1: Establish test foundation

**Problem:** No test files, no Vitest config, no Playwright config, no `e2e/` directory. 100% line coverage on changed code is mandated but unenforceable.

**Acceptance criteria:**
- `vitest.config.js` configured with v8 coverage provider; `npm run test:coverage` passes with 100% lines threshold on `api/_lib/`
- `playwright.config.js` configured targeting local dev server; `npm run test:e2e` executes at least one passing test
- `e2e/` directory created with a smoke test covering: load home page → select plan → select lot
- All fixes in this PRD include tests at the appropriate level

## Medium — Next quarter

### SEC-11: Add Content Security Policy headers

**Acceptance criteria:** All Vercel function responses include a `Content-Security-Policy` header allowing `self`, `fonts.googleapis.com`, and the Stripe JS domain; all other sources blocked.

---

### SEC-12: Enforce stronger admin password minimum

**Acceptance criteria:** `change-credentials.js` enforces minimum 12 characters; at least one uppercase, one digit, one special character; UI communicates requirements.

---

### BE-4: Enforce file type and size on authority doc uploads

**Acceptance criteria:** Server rejects non-PDF/image/DOCX MIME types; rejects payloads over 10 MB; client shows a clear error before submitting oversized files.

---

### BE-5: Add schema version to skip redundant migrations

**Acceptance criteria:** `tocs:data` (or replacement keys) include a `schemaVersion` field; `readData()` skips migration loop when version matches current; migration only runs once per deployment.

---

### BE-6: Reconcile server.js with Vercel API path

**Acceptance criteria:** `server.js` supports: Stripe checkout, keys product category, `keysShipping`, `orderCategory`, `applicantType`/`ownerName` in email, `logo` in config public response, `selectedShipping` in CSV export. Or `server.js` is explicitly deprecated and removed.

---

### BE-7: Harden Redis client reconnection

**Acceptance criteria:** Redis client registers an `on('error')` handler that resets `_client` to `null`; next invocation creates a fresh client rather than attempting to reconnect a broken instance.

---

### ARCH-2: Split [id]/[action].js into discrete handlers

**Acceptance criteria:** Each action (authority, status, send-certificate, send-invoice, stripe-confirm) is its own file or clearly isolated handler module; shared middleware extracted to `api/_lib/`.

---

### ARCH-3: Structured logging

**Acceptance criteria:** All log calls emit JSON with `{ level, requestId, orderId?, message, ...context }`; no bare `console.log` in production paths; request ID propagated from entry point through all async operations.

---

### ARCH-4: Move SharePoint upload before HTTP response

**Acceptance criteria:** SharePoint upload completes (or times out with a logged error) before `res.json()` is called; order record reflects SP upload success/failure; Vercel function does not freeze mid-upload.

---

### FE-3: Decompose App.jsx into components

**Acceptance criteria:** Each purchase flow step is a separate component file; admin panel is a separate component; shared UI primitives extracted; each component is independently testable with Vitest.

---

### FE-4: Add React ErrorBoundary

**Acceptance criteria:** Top-level `ErrorBoundary` component catches render errors and displays a user-friendly message with a reload option rather than a blank page.

---

### FE-5: Use cryptographically secure order ID (server-side, covered by SEC-5)

Already addressed under SEC-5.

---

### FE-6: Client-side file size validation before upload

**Acceptance criteria:** File input rejects files over 10 MB before the fetch call; user sees a clear error message.

---

### LEGAL-1: Immutable append-only audit log

**Acceptance criteria:** Audit log entries written to a separate Redis key (`tocs:audit:{orderId}`) using `RPUSH` (append-only list); never modified or deleted by any endpoint; admin delete marks order as `archived` in the orders store but does not touch audit keys.

---

### LEGAL-2: Soft delete for orders

**Acceptance criteria:** Admin delete sets `order.deletedAt` timestamp and `order.status = "Deleted"`; order is excluded from the customer-facing UI but retained in Redis and audit log; hard delete requires a separate explicit action and is logged.

---

### LEGAL-3: Create PRD files for all existing features

**Acceptance criteria:** `prd/order-flow.md` and `prd/admin-panel.md` created with user stories and acceptance criteria covering the current implemented behaviour; committed to `main`.

---

### A11Y-1 to A11Y-4: Accessibility baseline

**Acceptance criteria:**
- All icon-only buttons have `aria-label` attributes
- All form inputs have associated `<label>` elements via `htmlFor`/`id`
- Focus moves to new step content on step transitions
- Order submission and email sending status changes are announced via `role="alert"` or `aria-live="polite"`

---

### SRE-1: Health check endpoint

**Acceptance criteria:** `GET /api/health` returns `200 { status: "ok", redis: "ok" }` when Redis is reachable; returns `503` when Redis is unavailable; no auth required.

---

### SRE-2: Define SLOs and alerting

**Acceptance criteria:** SLOs documented (e.g. 99.9% uptime, p99 order placement < 3s); Vercel or external uptime monitor configured against `/api/health`; on-call contact defined in `docs/runbook.md`.
